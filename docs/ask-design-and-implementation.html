Skill Design Note

Ask · Skill 设计与实现说明

这是一份针对 ask 的设计与实现说明，目标是解释它为什么存在、解决什么问题、如何组织调用链，以及当前实现的扩展边界与使用约束。

结论先行：ask 不是一个“大而全”的代理框架，而是一个很薄的本地顾问调用层。 它把“选 provider、拼角色 prompt、执行本地 CLI、沉淀 artifact”这四步标准化，目的是让使用者能以更低心智成本复用外部模型视角。

1. 设计目标

ask 的目标不是替代主工作流，而是补一个稳定的“第二视角”入口。 当使用者已经在当前仓库内进行分析、排障或实现时，常常还需要一轮额外判断，例如代码 review、风险评估、架构取舍或 bug 根因假设。

因此，这个 skill 要满足三个设计目标：

  1. 本地执行。直接复用本机已经安装的 claude、gemini、codex CLI。
  2. 入口统一。无论 provider 是谁，用户都通过同一脚本入口发起请求。
  3. 结果可追溯。每次调用都落一个 markdown artifact，避免“问完即失忆”。

2. 适用场景与非目标

2.1 适用场景

  • 对当前改动做一轮外部 code review。
  • 对某个 bug 的根因给出第二视角分析。
  • 在多个实现方案之间做轻量架构判断。
  • 基于角色 prompt，让顾问从 reviewer、architect、debugger 等角度回答。

2.2 非目标

  • 不负责管理多轮复杂会话状态。
  • 不替代主 agent 的仓库读写流程。
  • 不负责远端 provider 路由，也不在本地 binary 缺失时静默降级。
  • codex provider 当前也不作为“直接改代码”入口，而是只读顾问入口。

3. 目录结构

ask/
├── SKILL.md
├── prompts/
│   ├── architect.md
│   ├── code-reviewer.md
│   └── debugger.md
└── scripts/
    └── ask.js

其中 SKILL.md 负责定义 skill 的触发语义和使用说明； prompts/ 负责角色化 prompt； scripts/ask.js 负责所有执行逻辑。

4. 执行链路

整个调用流程可以压缩成一条四段链路：

用户输入
    ↓
解析 provider / prompt / --agent-prompt
    ↓
调用本地 CLI（claude / gemini / codex）
    ↓
写入 .artifacts/ask/*.md
阶段 职责 对应实现
参数解析 识别 provider、prompt、角色 prompt 参数 parseArgs()
环境校验 确认本地 binary 存在 ensureBinary()
角色拼接 读取 prompts/<role>.md 并拼到最终 prompt 前 readAgentPrompt()
Provider 适配 为不同 CLI 生成不同参数 buildLaunchArgs()
产物沉淀 写入 markdown artifact writeArtifact()

5. Provider 适配策略

这个 skill 的核心不是“调用模型”本身，而是做一层很薄的 provider 适配。 当前适配表如下：

Provider 执行命令 设计意图
claude claude -p -- "<prompt>" 走 Claude CLI 的非交互 prompt 模式。
gemini gemini -p "<prompt>" 保持与 Gemini CLI 的最小参数集兼容。
codex codex exec --skip-git-repo-check --sandbox read-only "<prompt>" 限定为只读顾问路径，避免把 skill 演变成直接改工作树的入口。

这里最重要的判断是：codex 虽然可以非交互执行，但当前 skill 只把它定位成“问答 / review / 分析”顾问。 这能把使用边界收紧，让脚本行为可预测，也能降低误触写操作的风险。

6. 角色 Prompt 机制

ask 支持通过 --agent-prompt 指定角色文件。 角色文件本质是一个前置上下文块，会被拼接到用户 prompt 前面。 

role prompt

user prompt

当前内置了三类角色：

  • code-reviewer：偏 correctness、regression、readability。
  • architect：偏边界、依赖方向、扩展性与 trade-off。
  • debugger：偏可验证假设、状态差异、最小验证路径。

这个设计的好处是：脚本本身不必硬编码很多“回答模式”，而是把变化点下放到 prompt 文件。 新增角色时，只需要补一个新的 .md 文件，而不需要改脚本逻辑。

7. Artifact 设计

Artifact 是这个 skill 的核心价值之一。它确保每次咨询不是一次性输出，而是一个可复盘、可转发、可对照的记录。

当前 artifact 固定写入：

.artifacts/ask/<provider>-<slug>-<timestamp>.md

文件内容包括：

  1. Provider
  2. Exit code
  3. Created at
  4. Original task
  5. Final prompt
  6. Raw output
  7. Concise summary
  8. Action items

这种结构的优势在于，后续无论是人还是脚本，都能快速判断这次调用是否成功、实际发了什么 prompt、原始响应是什么，以及下一步是否需要继续落地。

8. 错误处理与边界控制

当前脚本采取的是“早失败、明报错”的策略：

  • provider 不合法，直接报 usage error。
  • prompt 缺失，直接报错。
  • 本地 binary 不存在，直接提示先用 --version 验证安装。
  • 角色文件不存在或为空，直接中止，而不是静默忽略。

这里的设计原则很简单：对一个顾问型脚本来说，最糟糕的情况不是失败，而是“看起来成功，实际上发错了 prompt 或用了错误 provider”。 因此，它宁可早失败，也不做隐式兜底。

9. 当前实现的扩展点

ask 现在已经能满足基础使用，但还有几个自然扩展点：

  • 为 codex 增加可选的 --model、--search、--output-last-message 透传。
  • 增加一个更顺手的仓库包装命令，例如 Scripts/ask。
  • 支持更细粒度的 artifact summary 生成，而不是固定成功模板。
  • 为常见工作流补更多角色 prompt，例如 ios-reviewer、test-engineer。

10. 使用示例

# 直接问 Claude
node ask/scripts/ask.js claude "请 review 当前改动"

# 问 Gemini
node ask/scripts/ask.js gemini "分析这个 bug 根因"

# 问 Codex
node ask/scripts/ask.js codex "请 review 当前实现风险"

# 带角色 prompt
node ask/scripts/ask.js codex --agent-prompt code-reviewer "请 review 当前重构"

11. 总结

ask 的价值不在于“封装了多少能力”，而在于它把一个高频但分散的动作标准化了： 使用者不再需要临时记忆不同 provider 的调用命令、角色 prompt 如何拼、结果该如何归档。

它是一个薄层工具，但正因为足够薄，才更容易保持稳定、易查、易扩展。 对于希望在日常开发中持续使用外部模型做二次确认的人来说，这种克制的实现反而更有长期价值。
